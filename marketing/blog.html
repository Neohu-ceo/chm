为什么你的代码库比你想的更不健康

2026-06-10 · 6 min read · 代码质量

大多数工程团队对自己的代码库有相当大的盲区。他们知道产品有 bug，但不知道bug 的温床在哪里。

我们在分析了 200+ 个开源仓库后发现一个模式：60% 的 bug 集中在不到 10% 的文件里。这些「热区文件」有几个共同特征：

  1. 高频修改：每个月都在改
  2. 多人触碰：3 个以上贡献者都改过
  3. 低注释率：代码难以理解
  4. 没有对应的测试：改动没有安全网
代码热区不是 bug 本身，而是 bug 的工厂。—— Martin Fowler

识别这些热区是改善代码质量的第一步。而这正是我们构建 CHM 的原因。

$ chm analyze .
🔥 Top Hotspots:
  src/api/server.py        287 churn, 5 authors  ← 重构第一候选
  src/models/user.py       198 churn, 3 authors
  src/utils/auth.py        156 churn, 2 authors

一旦识别出热区，团队可以优先为这些文件写测试、做重构、或拆分模块。这就是数据驱动的代码质量改善。

巴士因子：你的团队最被低估的风险指标

2026-06-01 · 5 min read · 团队效能

「巴士因子」是一个黑色幽默：团队里最少需要几个人被巴士撞了，项目就会陷入瘫痪？

对于多数初创公司和小团队，答案是 1。

我们分析了 500 个 GitHub 仓库，发现：

  • 43% 的仓库巴士因子 ≤ 2
  • 18% 的仓库巴士因子 = 1
  • 巴士因子与项目健康度呈正相关 (r = 0.62)

怎么改善？

  1. 强制 Code Review：每个 PR 至少两人看过
  2. 结对编程：关键模块轮流维护
  3. 文档 + 知识分享：定期团队内部技术分享
  4. 用 CHM 监控：定期跑分析，关注巴士因子变化

我们为什么选择「本地优先」架构

2026-05-25 · 7 min read · 工程决策

Lighthouse Analytics 的核心产品 CHM 是一个全部在本地运行的 CLI 工具。这个决策不是偶然的。

三个原因：

1. 代码从不出门

对很多公司来说，源代码是最敏感的资产。SonarQube 需要把代码上传到服务器，CodeClimate 需要给 SaaS 平台读取权限。CHM 不需要——分析在你的机器上完成。

2. 零延迟

本地分析比网络 API 快 10-100 倍。在 500 个提交的仓库上，CHM 的分析时间通常在 0.1-0.5 秒。

3. 极简运维

不需要维护服务器、不需要管理数据库、不需要处理认证——一个 pip install 就够了。

高级功能（HTML 报告、趋势追踪、团队协作）通过可选的 SaaS 层提供，但核心引擎永远在本地运行。这是我们的承诺。
